# vkmtl Checklist

This checklist tracks open decisions and implementation steps that should be
closed one by one as vkmtl moves from prototype to library.

## Already Clear

- [x] vkmtl is a graphics abstraction library, not a direct Vulkan/Metal union.
- [x] Backend modules must be pluggable and isolated.
- [x] Public examples belong under `examples/`.
- [x] The current triangle has become `examples/triangle`.
- [x] Examples should use the public vkmtl API, not raw Vulkan or Metal.
- [x] Slang is the shader source language.
- [x] Phase 0 starts with Vulkan and Metal bindings.
- [x] Public API style should be Metal-inspired where backend-neutral design
  allows it.

## Open Design Decisions

- [x] Choose the backend dispatch model.
  Options to evaluate: tagged union, function-table interface, or comptime
  backend specialization.

- [x] Define the first public module layout.
  Current layout centers on `src/vkmtl.zig`, `src/core.zig`, `src/runtime/`,
  `src/backend/vulkan/`, `src/backend/metal/`, and `src/shader/`. Windowing
  adapters live outside vkmtl core.

- [x] Define the public handle model.
  Decide whether `Device`, `Buffer`, `Texture`, `Pipeline`, and command objects
  are opaque structs, indices into backend-owned tables, or tagged backend
  wrappers.

- [x] Define ownership and lifetime rules.
  Decide which objects own allocators, which objects must be destroyed before
  their parent, and how debug builds report leaks or invalid destruction order.

- [x] Define backend selection precedence.
  Decide how `.auto`, explicit `.vulkan` / `.metal`, and environment overrides
  interact, including which error wins when multiple backends fail.

- [x] Define native-handle escape hatches.
  Decide naming, stability promises, and whether native handles are debug-only
  or part of the supported advanced API.

- [x] Define the platform surface boundary.
  Decide what belongs in core vkmtl versus external windowing adapters, and how
  macOS `CAMetalLayer` ownership is represented.

- [x] Define the minimum Metal bridge ABI for Phase 0.
  At minimum, decide whether Phase 0 needs only device probing or also command
  queue, library compilation, layer/drawable, and command buffer hooks.

- [x] Define shader tool dependencies.
  Decide whether `slangc` is discovered by `build.zig`, vendored, or provided
  through an explicit user path.

- [x] Define runtime shader cache artifact layout.
  Decide where generated SPIR-V, MSL, and reflection data live so examples and
  debugging can inspect them.

- [x] Define the first reflection format.
  Decide whether reflection data is generated by SPIR-V tools, shader compiler
  output, or manual descriptors until the pipeline is stable.

- [x] Define Slang binding rules.
  Decide how Slang resource declarations and explicit binding annotations map
  to vkmtl bind groups and backend-specific bindings.

- [x] Define the first `examples/` layout.
  Decide names and build steps for `examples/triangle`, shader files, assets,
  and shared example helpers.

- [x] Define validation commands per phase.
  Decide the expected command set for docs-only edits, API edits, backend probe
  edits, and rendering behavior edits.
